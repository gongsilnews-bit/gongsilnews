# 💡 IDEA 11: 회원(Members) 데이터 아키텍처 설계 회의록

본 문서는 공실뉴스 플랫폼의 회원 데이터 구조를 설계하기 위해 진행된 기획 회의 내용을 정리한 것입니다.

---

## 1. 회원구분은 딱 3가지 (Role)

복잡하게 6~7개로 나누지 않고, **역할(role)은 3가지만** 만든다. 세부 구분은 역할을 늘리지 않고 다른 컬럼(칸)의 값으로 처리한다.

| role | 명칭 | 설명 |
|------|------|------|
| `ADMIN` | 관리자회원 | 플랫폼 전체 관리 (최고관리자 + 부관리자) |
| `REALTOR` | 부동산회원 | 매물 등록, 서브도메인 관리 (소장 + 직원) |
| `USER` | 일반회원 | 뉴스 열람, 찜하기 (무료 + 유료) |

### 세부 구분은 role을 늘리지 않고 옆 칸(컬럼)으로 처리

| 원래 구분 | role | 세부 구분 방법 |
|----------|------|-------------|
| 최고관리자 | `ADMIN` | `appointed_by`가 비어있음 (아무도 지정 안 함 = 최고) |
| 부관리자 | `ADMIN` | `appointed_by`에 최고관리자 ID가 들어있음 |
| 부동산 소장 | `REALTOR` | `agencies.owner_id`가 내 ID |
| 부동산 직원 | `REALTOR` | `agency_members`에 소속되어 있으나 owner가 아님 |
| 일반회원 | `USER` | `plan = FREE` |
| 일반유료회원 | `USER` | `plan = PAID_1` 또는 `PAID_2` |

### 왜 3가지로 충분한가?
- role을 6개로 만들면 → 코드에서 `role === 'REALTOR_FREE' || role === 'REALTOR_PAID1' || ...` 이런 지저분한 조건이 생김
- role 3개 + plan 분리하면 → `role === 'REALTOR'` 한 줄이면 끝. 요금제는 따로 체크

---

## 2. 요금제(Plan)는 역할과 분리

**"누구인가(role)"와 "돈을 내는가(plan)"는 완전히 독립적으로 움직인다.**

| 대상 | plan 저장 위치 | 이유 |
|------|-------------|------|
| 부동산 | `agencies.plan` | 소장이 업체 단위로 결제. 직원은 공짜로 따라감 |
| 일반회원 | `members.plan` | 개인이 본인 것만 결제 |

### 동작 예시
- 부동산 소장이 유료 가입 → `agencies.plan = 'PAID_1'` → 직원 전부 유료 기능 사용
- 일반회원이 유료 구독 → `members.plan = 'PAID_1'` → 본인만 유료 기능 사용
- 요금제 변경 시 → plan 값만 수정. role은 건드리지 않음

## 3. 3개 테이블 구조

### 3-1. `members` — 회원 마스터 테이블

모든 회원(관리자, 부동산, 일반)의 기본 정보를 보관하는 핵심 테이블.

| 컬럼 | 설명 |
|------|------|
| `id` | 회원 고유번호 (auth.users와 1:1 연결) |
| `email` | 이메일 |
| `name` | 이름 |
| `phone` | 연락처 |
| `avatar_url` | 프로필 사진 |
| `provider` | 가입경로 (google / kakao / naver) |
| `role` | **회원구분 3가지: ADMIN / REALTOR / USER** |
| `plan` | 요금제 (FREE / PAID_1 / PAID_2) — 일반회원용 |
| `appointed_by` | 누가 지정했나 (부관리자 구분용) |
| `status` | 상태 (active / suspended) |
| `signup_completed` | 가입 마무리 모달 완료 여부 |
| `created_at` | 가입일 |
| `updated_at` | 수정일 |

### 3-2. `agencies` — 부동산 업체 테이블

부동산 업체 정보를 보관. 서브도메인, 테마, 업체 과금의 근거가 되는 B2B 핵심 허브.

| 컬럼 | 설명 |
|------|------|
| `id` | 업체 고유번호 |
| `owner_id` | **소장 회원 ID** (이 값으로 소장/직원 구분) |
| `name` | 상호명 |
| `ceo_name` | 대표자명 |
| `phone, cell` | 대표전화 / 휴대폰 |
| `zipcode, address, address_detail` | 주소 |
| `lat, lng` | 위경도 (On-Write Geocoding) |
| `biz_num, reg_num` | 사업자등록번호 / 중개등록번호 |
| `biz_cert_url` | 사업자등록증 파일 |
| `subdomain` | 서브도메인 슬러그 (예: happy → happy.gongsilnews.com) |
| `theme_id` | 선택한 테마 |
| `logo_url` | 커스텀 로고 |
| `plan` | **업체 요금제** (FREE / PAID_1 / PAID_2) — 소장이 결제, 직원 전부 적용 |
| `status` | 상태 (ACTIVE / SUSPENDED / DELETED) |

### 3-3. `agency_members` — 소속 관계 + 직원별 권한 설정

소장이 직원별로 세밀하게 권한을 조절하는 공간.

| 컬럼 | 설명 |
|------|------|
| `agency_id` | 어느 부동산 |
| `member_id` | 어느 직원 |
| `is_owner` | 소장 여부 (TRUE / FALSE) |
| `max_properties` | 매물 등록 한도 (NULL이면 무제한) |
| `max_articles` | 기사 등록 한도 |
| `can_view_stats` | 통계 열람 가능 여부 |
| `can_manage_board` | 게시판 관리 가능 여부 |
| `note` | 소장 메모 |

## 4. agencies 테이블이 반드시 필요한 이유

### 4-1. 소장 교체 시 안전성
- 매물이 "사람(user_id)"에 붙어있으면 → 소장 탈퇴 시 매물 미아됨
- 매물이 "업체(agency_id)"에 붙어있으면 → 소장만 교체, 매물은 그대로 유지
- 비유: 개인 명의 사업(위험) vs 법인 사업(안전)

### 4-2. 네트워크 광고 (연락처 마스킹)
- A부동산이 매물 등록 후 [네트워크광고 ON] 클릭
- B부동산 서브홈에서 같은 매물이 보이되, **B부동산 이름/연락처로 교체**되어 노출
- 매물 데이터는 1건만 저장, 보여줄 때만 이름을 바꿔치기 (속도 영향 거의 없음)

### 4-3. 서브도메인 시스템의 기반
- `agencies.subdomain = 'happy'` → `happy.gongsilnews.com`
- `agencies.theme_id` → 홈페이지 테마 변경
- 최고관리자가 업체의 상태를 통제 (활성/정지/삭제)

## 5. 테이블 간 연결(관계) 구조

```
members (회원)
  │
  ├── articles.author_id     → 이 회원이 쓴 기사
  ├── properties.user_id     → 이 회원이 입력한 매물
  │
  └── agency_members          → 이 회원이 소속된 부동산
        │
        └── agencies           → 부동산 업체 정보
              │
              ├── properties.agency_id  → 이 업체 소속 매물 (소유권)
              └── owner_id → members.id  → 이 업체의 소장
```

핵심: **매물에는 `user_id`(누가 입력)와 `agency_id`(어느 업체 소속) 두 개가 모두 필요.**
- `user_id` = 기록용 (누가 입력했는지)
- `agency_id` = 소유권 (어느 업체 것인지)

## 6. DB 개념 정리 (회의 중 논의 사항)

### 테이블 vs 컬럼 vs 로우
| 용어 | 엑셀 비유 | 예시 |
|------|----------|------|
| 데이터베이스 | 엑셀 파일 1개 | 공실뉴스 DB 전체 |
| 테이블 | 시트 탭 | members, properties, articles |
| 컬럼 | 세로 열 (항목) | 이름, 이메일, 가격, 주소 |
| 로우 | 가로 행 (데이터 1건) | 김소장의 정보 한 줄 |

### 속도에 영향을 주는 것 / 안 주는 것
| 항목 | 속도 영향 |
|------|----------|
| 테이블 개수 (26개) | ❌ 영향 없음 (1000개도 OK) |
| 한 테이블에 데이터 50만 건 | ⚠️ 인덱스 필요 |
| 컬럼 200개 (한 테이블) | ⚠️ 설계 문제 → 30~50개가 적정 |
| 배열(ARRAY) 안에 데이터 200개 | ✅ 영향 없음 |

### 검색(필터)용 vs 표시용 컬럼
- **검색에 쓰는 항목** (가격, 지역, 유형) → 각각 컬럼으로 분리 (인덱스 가능)
- **보여주기만 하는 항목** (옵션 목록, 이미지) → 배열/JSONB 하나에 묶어도 됨
- 배열 안에서도 검색 가능: `WHERE '에어컨' = ANY(options)` + GIN 인덱스

| 테이블 | 데이터 | 처리 |

- `articles` (24개 컬럼) — author_id로 회원 연결 구조 갖춤
- `boards`, `board_posts`, `comments` 등 — 정상

## 8. 자동 동기화 (소셜 로그인 → 회원 자동 생성)

소셜 로그인 성공 시 `auth.users`에 레코드가 생기면, 트리거(자동장치)가 `members`에 기본 정보를 복제한다.

```
소셜 로그인 → auth.users INSERT → 트리거 동작 → members INSERT (signup_completed = FALSE)
→ 가입 마무리 모달 표시 → 추가정보 입력 → members UPDATE (signup_completed = TRUE)
```

---

## 9. 향후 작업 순서 (확정 시)

1. 기존 `members`, `profiles`, `realtor_info` 테이블 삭제
2. 새 `members` 테이블 생성 (role 3가지 + plan 분리)
3. `agencies` 테이블 생성 (부동산 업체 정보)
4. `agency_members` 테이블 생성 (소속 + 직원별 권한)
5. `auth.users → members` 자동 동기화 트리거 생성
6. `properties` 테이블에 `agency_id` 컬럼 추가
7. RLS(보안 정책) 설정
8. 가입 마무리 모달(`SignupCompleteModal.tsx`) DB 연결
9. Header 로그인 상태 표시

## 10. 인덱스(Index) — DB 속도의 핵심

### 인덱스란?
**전화번호부의 "ㄱㄴㄷ 탭"과 같은 원리.** DB가 데이터를 빨리 찾기 위해 미리 만들어놓는 목차.

```
❌ 인덱스 없음 = 1만 명 전화번호부에서 "김소장"을 1쪽부터 다 뒤짐 → 느림
✅ 인덱스 있음 = "ㄱ → 23쪽" 목차 보고 바로 점프 → 빠름
```

### 인덱스 vs 컬럼 차이
- **컬럼** = 데이터 자체 (이름, 전화번호 등). 없으면 데이터가 없음 (필수)
- **인덱스** = 그 데이터를 빨리 찾는 목차 (부가장치). 없어도 동작하지만 느릴 뿐 (선택)

### 인덱스 특징
- SQL 한 줄로 걸 수 있음: `CREATE INDEX idx_지역 ON properties (sigungu);`
- 한번 걸면 데이터 추가/수정/삭제 시 **자동으로 목차 업데이트**
- 지금은 데이터가 적어서 불필요. **나중에 느려지면 그때 추가하면 됨**

### 인덱스가 필요한 시점
| 로우(데이터) 수 | 인덱스 필요 여부 |
|---------------|---------------|
| 1만 건 | 없어도 빠름 |
| 10만 건 | 있으면 좋음 |
| 100만 건 | 반드시 필요 |

## 11. Supabase 플랫폼 특징 정리

Supabase = **서버 개발자 없이도 서비스를 만들 수 있게 해주는 백엔드 올인원 도구**

### 핵심 기능 6가지
| # | 기능 | 공실뉴스 활용 |
|---|------|-------------|
| ① | **DB (PostgreSQL)** | 회원, 매물, 기사 저장 |
| ② | **Auth (인증)** | 구글/카카오 소셜 로그인 |
| ③ | **Storage (파일)** | 매물 사진, 사업자등록증 업로드 |
| ④ | **Realtime (실시간)** | 카톡 같은 채팅, 쪽지 실시간 알림 |
| ⑤ | **RLS (보안)** | 내 매물은 나만 수정 가능 |
| ⑥ | **Edge Functions** | AI 기능 등 서버 로직 |

### 현재 요금제 (Pro)
| 항목 | 용량 | 여유도 |
|------|------|--------|
| DB 용량 | 8GB | 매물 수십만 건 OK |
| 파일 저장 | 100GB | 사진 수만 장 OK |
| 리얼타임 연결 | 500개 동시 | 회원 간 쪽지 여유 |
| 회원 1만 명 | 약 5MB | 전혀 문제없음 |

### Supabase를 선택한 이유 (vs MySQL 등)
| 기능 | PostgreSQL (Supabase) | MySQL |
|------|---------------------|-------|
| 지도 기능(PostGIS) | ✅ 최강 | ⚠️ 기본만 |
| 배열(ARRAY) 지원 | ✅ 기본 내장 | ❌ 없음 |
| JSONB 검색 | ✅ 빠름 | ⚠️ 느림 |
| 실시간(Realtime) | ✅ 내장 | ❌ 별도 구축 |

**프롭테크(부동산 기술) 서비스에는 PostgreSQL이 업계 표준** (직방, 다방 등도 사용)

## 12. PostGIS — 지도 매물 검색 아키텍처

### PostGIS란?
PostgreSQL에 내장된 **지도 전용 엔진**. 좌표(lat, lng) 기반의 공간 계산을 DB가 직접 처리해줌.

### 동작 흐름
```
[매물 등록 시]
중개사가 주소 입력 → 카카오 API가 좌표 변환 → lat, lng를 DB에 저장 (1번만)

[손님이 지도를 볼 때]
지도 화면의 네모 좌표를 DB에 보냄 → PostGIS가 그 안의 매물만 골라서 돌려줌

[지도를 이동하면]
새로운 네모 좌표를 다시 보냄 → 해당 영역 매물만 다시 가져옴 (자동 반복)
```

### BBOX (Bounding Box) 방식
```
사용자 화면에 보이는 영역:
  좌하단 좌표 (37.49, 127.01) ~ 우상단 좌표 (37.52, 127.07)

DB 쿼리:
  5만 개 매물 중 이 네모 안에 있는 것만 → 150개만 돌려줌
  나머지 49,850개는 아예 안 가져옴 → 속도 극한 유지
```

### 클러스터링 (핀 묶기)
```
지도를 많이 축소하면 핀 수천 개가 겹침 → 화면 멈춤 위험

해결: 여러 핀을 하나로 묶어서 숫자로 표시
  [강남 1,500] [서초 980] [마포 720]
  숫자를 클릭하면 확대되면서 개별 핀이 펼쳐짐

이 묶기 계산도 PostGIS가 DB에서 처리 → 브라우저 부하 없음
```

### PostGIS 활용 범위 (매물만이 아님)
| 기능 | 어떤 테이블 | 활용 |
|------|-----------|------|
| 공실열람 지도 | properties (lat, lng) | 화면 내 매물만 표시 |
| 우리동네뉴스 | articles (lat, lng) | 내 위치 근처 기사만 표시 |
| 주변 부동산 찾기 | agencies (lat, lng) | 내 주변 중개사무소 표시 |
| 지역 기반 광고 | Geo-Fencing | 매물 반경 3km 부동산에만 네트워크 광고 노출 |

## 13. 실시간 채팅 기능 (Supabase Realtime 활용)

### 공실뉴스 내부 채팅 = 카톡 대체
```
기존: 손님이 매물 봄 → 전화번호 복사 → 카톡 열기 → 대화 (사이트 이탈 😩)
채팅: 손님이 매물 봄 → [문의하기] 클릭 → 바로 채팅 시작 (사이트 안에서 😊)
```

### 필요한 테이블 2개 추가
```
chat_rooms (채팅방)
  ┌──────────┬─────────────────────────┐
  │ 채팅방 ID │ 참여자                    │
  ├──────────┼─────────────────────────┤
  │ room_001 │ 임대인 ↔ 김소장            │ ← 1:1 채팅
  │ room_002 │ 김소장 ↔ 박소장 ↔ 이소장    │ ← 단체 채팅
  └──────────┴─────────────────────────┘

chat_messages (메시지)
  ┌──────────┬────────┬──────────────┬───────┐
  │ 채팅방 ID │ 보낸사람 │ 내용          │ 시간   │
  ├──────────┼────────┼──────────────┼───────┤
  │ room_001 │ 임대인  │ 방 볼수있나요?  │ 12:19 │
  │ room_001 │ 김소장  │ 내일 2시요!    │ 12:20 │
  └──────────┴────────┴──────────────┴───────┘
```

### 구현 가능한 기능
- ✅ 1:1 채팅 (임대인 ↔ 부동산)
- ✅ 단체 채팅방 (부동산끼리 공동중개 논의)
- ✅ 읽음/안읽음 표시
- ✅ 새 메시지 알림 (빨간 숫자 뱃지)
- ✅ 사진/파일/서류 전송 (Supabase Storage 활용)
- ✅ 매물 카드 공유 (매물 링크 보내면 미리보기 표시)

### 다양한 채팅 시나리오
```
임대인 ↔ 부동산    "방 볼 수 있나요?"
손님   ↔ 부동산    "가격 협상 가능한가요?"
부동산 ↔ 부동산    "이 매물 공동중개 하실래요?"
관리자 → 전체 공지  "시스템 점검 안내"
```

### 비즈니스 효과
- 카톡으로 나가면 → 손님이 사이트를 떠남 → 재방문 불확실
- 공실뉴스 안에서면 → 손님이 계속 머무름 → 다른 매물도 보게 됨
- **사이트 체류 시간 증가 → 광고 수익 증가 → 플랫폼 가치 증가**
