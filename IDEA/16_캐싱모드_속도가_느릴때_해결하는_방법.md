# 16. 캐싱 모드 (Next.js & Vercel) 속도가 느릴 때 해결하는 방법

본 문서는 Next.js(App Router) 프로젝트를 Vercel에 배포했을 때, 로컬(개발 환경)에서는 빠르지만 실제 배포 환경에서 발생하는 **초기 로딩 딜레이(TTFB 지연) 문제**를 해결하기 위한 핵심 트러블슈팅 가이드입니다. 

Vercel 환경에서의 렌더링 병목 현상을 타개하고 극한의 속도(Zero-latency)를 끌어내기 위한 아키텍처 개편 내용을 빠짐없이 기록합니다.

---

## 🚨 문제 상황: "메인 페이지 렌더링이 레거시보다 느려 보임"

`export const dynamic = 'force-dynamic';` 셋팅을 사용하여 매 랜더링 시 실시간으로 DB에서 데이터를 가져오는 페이지(예: 메인 홈페이지, 공실열람 지도)가 다음과 같은 속도 저하를 발생시킵니다.

### 1-1. 병목 원인 3요소
1. **Serverless 콜드 스타트 (서버 부팅 지연)**
   - 개인 PC(로컬)와 달리 Vercel의 무료/서버리스 환경은 요청이 없으면 잠을 잡니다. 
   - 새로운 사용자가 접속하면 서버 런타임을 부팅해야 하므로 약 1~2초의 순수 대기 시간이 유발됩니다.
2. **물리적 네트워크 Latency (태평양 왕복 속도)**
   - Vercel의 기본 함수 실행 지역(예: 미국 D.C)과 실제 DB(한국 Supabase 리전)가 다를 경우, 매 요청마다 물리적 거리로 인한 통신 패널티(200~300ms)가 발생합니다.
3. **데이터 무한 대기 (force-dynamic의 맹점)**
   - 지도 핀 정보, 기사 목록 등 대량의 실시간 데이터를 DB에서 **모두 가져올 때까지 렌더링 자체를 멈추고 기다리는 방식**이기 때문에 하얀 백지 화면으로 머무는 딜레이(Waterfall)를 초래합니다.

---

## 💡 해결책: ISR (Incremental Static Regeneration) 캐싱 전략

매 접속마다 쌩-서버를 갈구는 `force-dynamic` 대신, Vercel의 CDN망에 미리 그려진 HTML을 올려두고 꺼내먹는 전략(ISR)으로 우회해야 합니다.

### 2-1. `force-dynamic` ➡️ `revalidate` 속성으로 변경
각 페이지 최상단의 dynamic 족쇄를 풀고, 특정 시간(예: 60초)마다 한 번씩만 캐시를 갱신하도록 명령을 변경합니다.

```tsx
// ❌ 기존 (느림: 매번 무조건 새로 DB 긁어옴)
// export const dynamic = 'force-dynamic';

// 🟢 변경 (초고속: 60초마다 갱신된 CDN 캐시를 0.01초 만에 제공)
export const revalidate = 60;
```
이렇게 하면 최초 1회 방문자를 제외한 모든 후속 방문자는 0.01초 만에 Vercel Edge Node(가장 가까운 캐시 서버)에서 미리 준비된 지도를 띄울 수 있습니다.

---

## 🔎 주의사항: "분명 revalidate 60을 줬는데 왜 아직도 느리죠?"

Next.js 빌드 창을 열었을 때, 여전히 `ƒ (Dynamic)` 마크가 찍히며 캐시(`○ Static`)가 적용되지 않는 황당한 변수를 조심해야 합니다. 
페이지 단에서 아무리 캐시 명령을 내려도, **하위 함수 중 단 한 곳이라도 강력한 '캐시 파괴(Opt-out)' 족쇄**를 차고 있으면 해당 페이지 전체가 암묵적으로 `Dynamic`으로 강제 다운그레이드됩니다.

### 3-1. Supabase Client 내장 Fetch의 'no-store' 족쇄 해제
실제 공실뉴스 이슈의 원인이었던 부분입니다. DB 통신 라이브러리 내부에 엮인 속성을 풀어야 합니다.

**`src/app/actions/vacancy.ts` 안의 족쇄 제거 예시:**
```typescript
// ❌ 캐시를 원천 차단하는 범인 코드
function getAdminClient() {
  return createClient(supabaseUrl, serviceKey, {
    auth: { autoRefreshToken: false, persistSession: false },
    global: {
      fetch: (url, options) => {
        // 이 cache: 'no-store' 구문 하나 때문에 Next.js의 모든 ISR 캐싱이 무효화됨!
        return fetch(url, { ...options, cache: 'no-store' });
      }
    }
  });
}

// 🟢 해결: global fetch 덮어쓰기를 삭제하여 Next.js의 기본 캐시 체계(revalidate)를 따르게 함
function getAdminClient() {
  return createClient(supabaseUrl, serviceKey, {
    auth: { autoRefreshToken: false, persistSession: false },
  });
}
```

### 3-2. 다른 발생 가능 변수 (Dynamic 강제 전환 요소)
- `useSearchParams()`를 클라이언트 컴포넌트에서 빈번하게 사용할 경우 (단, `<Suspense>`로 묶어두면 방어 가능)
- 컴포넌트 내부에서 구버전의 `cookies()`나 `headers()`를 서버 사이드에서 직접 까서 읽어볼 경우 (사용자 고유 세션이 필요하므로 캐시 불가 판정)

## ✅ 최종 결과 검증 방법

터미널에서 `npm run build`를 실행했을 때 나오는 결과창(Route)을 반드시 확인해야 합니다.

```text
Route (app)             Revalidate  Expire
┌ ○ /                           1m      1y   (✅ 메인 홈페이지: 60초 캐싱 성공!)
├ ○ /gongsil                    1m      1y   (✅ 공실열람: 60초 캐싱 성공!)
├ ○ /news_map                   1m      1y   (✅ 우리동네뉴스: 60초 캐싱 성공!)
...
○  (Static)   prerendered as static content
ƒ  (Dynamic)  server-rendered on demand
```
모든 무거운 지도를 거느린 페이지들이 `ƒ(Dynamic)`에서 **`○(Static, 1m)`** 로 온전히 격상되었다면 캐싱 적용이 완벽하게 끝난 것입니다. Vercel 배포 후 테스트해 보면 레거시 시스템보다도 지연 없이 즉각적으로 화면이 뜨는 극한의 속도를 맛볼 수 있습니다.
