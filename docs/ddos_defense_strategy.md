# 공실뉴스 DDoS 방어 및 서버 인프라 안정성 전략

본 문서는 서비스 공식 런칭 전후로 발생할 수 있는 DDoS(디도스) 공격 및 악성 트래픽 폭주에 대비하기 위한 아키텍처 보호 전략을 정리한 문서입니다.

## 1. 아키텍처 개요 및 잠재적 취약점

현재 공실뉴스의 서비스 인프라는 다음과 같이 구성되어 있습니다:
- **코드 저장소:** GitHub
- **프론트엔드/웹 서버:** Vercel (Next.js)
- **데이터베이스:** Supabase (PostgreSQL)

이러한 모던 서버리스(Serverless) 스택은 전통적인 서버와 다른 형태의 취약점을 가집니다.

*   **Vercel (웹 서버)의 요금 폭탄 리스크:** Vercel은 트래픽이 급증할 때 서버가 다운되는 대신 자동으로 무한 확장(Auto-scaling)됩니다. 따라서 서비스는 유지되지만, 악성 봇이 수천만 번의 요청을 보낼 경우 막대한 클라우드 요금이 청구될 수 있습니다.
*   **Supabase (DB)의 병목 현상:** Vercel은 버티더라도, 해당 트래픽이 DB 조회(예: 공실 매물 검색)로 이어지면 Supabase에 연결 과부하가 발생하여 DB가 다운될 수 있습니다. DB가 마비되면 정상적인 유저들도 서비스를 이용할 수 없게 됩니다.
*   *참고:* 로컬 코드를 푸시하는 GitHub는 철저히 개발 파이프라인의 일부일 뿐입니다. 해커가 GitHub를 공격하더라도 이미 Vercel에 배포되어 돌아가고 있는 라이브 웹사이트에는 아무런 영향을 주지 못합니다.

## 2. 3단계 방어 전략

이러한 취약점을 막기 위해 다음과 같은 3단계 방어선을 구축합니다.

### 🛡️ 1차 방어선: 최전방 트래픽 차단 (Cloudflare)
*   **도메인 네임서버 우회:** `gongsilnews.com` 도메인을 Vercel로 직접 연결하지 않고, 전 세계 1위 CDN/보안 업체인 **Cloudflare**를 프록시로 거치도록 설정합니다.
*   **묻지마 방어:** Cloudflare의 AI가 악성 봇이나 디도스 트래픽을 감지하여 Vercel에 도달하기 전에 차단합니다.
*   **비용 대비 효과:** 무료 요금제(Free)만으로도 99% 이상의 일반적인 디도스 공격을 방어할 수 있으며, 서버 캐싱을 통해 Vercel 비용을 획기적으로 절감하고 페이지 로딩 속도를 향상시킵니다. (필요시 월 $20 수준의 Pro 요금제로 업그레이드)

### ⚔️ 2차 방어선: 애플리케이션 레벨 (Next.js 서버)
*   **Rate Limiting (접속 속도 제한):** 로그인, 회원가입, 매물 검색 API 등에 `Upstash (Redis)`를 활용한 Rate Limit을 적용합니다. (예: 1초에 5번 이상 검색 시 1분 차단)
*   **봇 차단 (CAPTCHA):** 중요 폼 입력 시 Cloudflare Turnstile이나 reCAPTCHA를 도입하여 봇의 자동 입력을 막습니다.

### 🏰 3차 방어선: 데이터베이스 보호 (Supabase)
*   **커넥션 풀링(Connection Pooling):** 트래픽 스파이크 발생 시 DB가 터지지 않도록 Supabase에서 제공하는 `PgBouncer` 또는 `Supavisor`를 적극 활용합니다.
*   **RLS (행 수준 보안):** 권한이 없는 사용자가 DB의 전체 데이터를 강제로 크롤링(Dump)해가는 것을 막기 위해 이미 설정된 RLS 정책을 유지 및 강화합니다.

## 3. 오픈 전 필수 액션 아이템

안전하고 성공적인 런칭을 위해 다음 사항을 우선적으로 설정해야 합니다.

1. **Cloudflare 도메인 연동 (우선순위: 매우 높음)**
   - Cloudflare에 가입 후 사이트를 추가하고, 도메인 구입처(가비아, 카페24 등)에서 네임서버를 Cloudflare가 제공한 주소로 변경합니다.
2. **핵심 API Rate Limit 적용 (우선순위: 높음)**
   - 매물 검색 및 인증 관련 서버 액션에 1차적인 속도 제한 로직을 추가하여 DB 과부하를 예방합니다.
