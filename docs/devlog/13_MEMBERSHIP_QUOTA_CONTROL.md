# 부동산 회원 요금제 및 이용 한도 제어 시스템 구현

**Date:** 2026-04-21

## 📌 개요
부동산 회원(REALTOR) 등급을 요금제(무료, 공실뉴스, 공실등록) 단위로 세분화하고, 관리자가 **매물 등록 수(총 건수)**와 **뉴스 기사 작성 수(월 단위)** 한도를 개별 및 일괄로 강력히 제어할 수 있는 시스템 방어선과 사용자 인터페이스(UI)를 구축하였습니다.

## 🛠 주요 구현 내용

### 1. 강력한 서버 측 2중 한도/만료 검증 (Fallback 방식)
프론트엔드에서의 UI 가림막을 넘어, API(서버 액션) 단에서 우회를 원천 차단하는 로직을 구비했습니다.
- **`src/utils/planCheck.ts`**: 현재 서버 시간과 회원이 구독하고 있는 요금제의 `plan_end_date`를 조회하여 만료 여부를 판독합니다. 기한이 단 1초라도 지났을 경우 무조건 `free` 등급으로 강등시켜 이후 기능들을 제약합니다. (`admin` 등급은 항상 프리패스)
- **`src/app/actions/vacancy.ts`**: DB 쿼리를 통해 본인이 새로 매물 카드를 생성할 때, 현재 등록한 매물 갯수(COUNT)를 조회하여 `max_vacancies` 초과 여부를 실시간 검열합니다.
- **`src/app/actions/article.ts`**: 요금제가 '공실뉴스부동산'인지 확인하고, 현재 날짜부터 **이번 달 1일 기준까지의 작성 건수**를 실시간 카운트(`createdAt` 기반)하여 `max_articles_per_month` 초과 여부를 엄격히 계산해 차단합니다. (배치 스케줄러 불필요)

### 2. 회원 핀셋(개별) 조절 관리자 폼 추가
- **`src/components/admin/MemberRegisterForm.tsx`**: 
  - 특정 회원의 [수정] 버튼 클릭 시, 해당 회원만의 요금제, 기간, 매물 한도, 기사 방어 한도를 직접 Number 값으로 조절할 수 있는 구역을 신설했습니다.
  - 이를 통해 특정 우수 제휴점 등에만 파격적인 한도를 무한대에 가깝게 독자 제공할 수 있습니다.

### 3. 회원 대량 일괄 갱신 (Bulk Update) 모달 및 액션 추가
- **`src/components/admin/sections/MemberSection.tsx`**: 
  - 회원 목록 좌측 체크박스를 통해 다중 체크 기능을 활성화하였습니다.
  - 상단 **`[요금제 일괄변경]`** 버튼 클릭 시 전용 팝업이 뜨며, 여러 명을 단번에 동일한 요금제/만료일/매물수/기사수로 업데이트 할 수 있도록 하여 영업부서의 운영 효율성을 최적화했습니다.
- **`src/app/admin/actions.ts`**: 
  - `adminBulkUpdatePlanAndLimits`라는 전용 대량 `UPDATE` 지원 액션을 신설하여 서버 DB단에서 안전하게 일괄 매핑이 이뤄지도록 하였습니다.

### 4. 회원 목록 리스트 데이터 및 컬럼 확장
- **`MemberSection.tsx` 리스트 개편**:
  - 기본 정보(아이디, 이름 등)를 좌측으로 재배치하고 우측에 운영 데이터 컬럼들을 추가하였습니다.
  - **공실 현황**: `현재 등록수 / 총 한도` 표시.
  - **기사 현황(월)**: `이번 달 작성수 / 월 한도` 표시.
  - **홈페이지 ID**: 제휴사별 배정된 도메인/Tenant ID를 리스트에서 즉시 확인 가능.
  - **종료일**: 유료 요금제 만료 날짜를 명시적으로 표기.

### 5. 승인 상태 자동 전환 로직 적용
- **정상승인 -> 정상 자동 전환**:
  - 부동산 회원이 승인된 후 7일(1주일)이 경과하면, 강조된 '정상승인' 뱃지 대신 차분한 '정상' 텍스트로 자동 전환되도록 로직을 구현했습니다.
  - 이를 통해 관리자가 최근 승인된(케어가 필요한) 신규 회원과 기존 정착 회원을 시각적으로 쉽게 구분할 수 있습니다.

## 🚀 향후 확장 제안
현재 기본값인 매물 5건, 기사 0건 한도에 대환 '초기 디폴트 세팅값' 제어는 로직/스키마 단에 설정되어 있습니다. 추후 `설정(Settings)` 관리자 화면을 통해 글로벌 운영 정책 데이터로 관리 테이블을 완전히 빼는 방식의 고도화 작업이 권장됩니다.
