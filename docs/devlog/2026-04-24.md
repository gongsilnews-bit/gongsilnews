# 2026-04-24 개발 일지 (Devlog)

## 1. 관리자 및 회원 관리 기능 버그 수정 및 고도화

### 부동산 회원 권한 변경 버그 수정
- **문제점**: 부동산 회원(REALTOR) 본인이 마이페이지(정보설정, `MemberRegisterForm`)에서 프로필을 수정할 때, '회원구분' 필드가 `<select>` 드롭다운 형태로 노출되었습니다. 이로 인해 마우스 휠 스크롤이나 실수로 인한 클릭 시 해당 값이 '일반회원'으로 변경되는 문제가 발생했습니다. 상태가 변경되는 즉시 부동산 관련 정보 탭 전체가 화면에서 사라지는 치명적이고 혼란을 주는 UI 버그가 있었습니다.
- **해결책**: 최고관리자(`isAdmin === true`)가 아닌 일반 사용자나 부동산 회원이 정보설정 페이지 접근 시, 회원구분 필드를 수정 불가능하도록 완전히 고정했습니다. 기존의 선택 가능한 박스 대신 직관적인 **파란색 텍스트 뱃지 (예: 부동산회원)** 형태로 렌더링되게 하여 문제를 구조적으로 차단했습니다.

### 서버 액션 디버깅 강화
- `actions.ts` 내 `adminGetMemberDetail` API에서 회원 상세 데이터 로드 실패 시, 디버깅이 어렵던 문제를 보완하기 위해 캐치 블록과 에러 조건에 서버 로깅(`console.error`)을 추가했습니다.

## 3. 서류 재확인 및 변경 알림 시스템 고도화
- **핵심 정보 변경 시 PENDING 강등 (재검수 시스템)**: 한 번 `APPROVED`(정상승인) 판정을 받은 부동산 회원이더라도, 마이페이지(정보설정)에서 주소, 상호명, 등록번호를 변경하거나, 새로운 등록증/사업자등록증을 업로드하고 '저장' 버튼을 누르면 시스템이 스스로 이를 감지하여 **자동으로 '승인대기' 상태로 전환**하는 기능을 도입했습니다. 악용(서류 바꿔치기)을 막고, 최고관리자의 정상적인 승인 플로우를 거치게 기획되었습니다. 다만, 사용 불편이 없게 기존 매물 노출 권한은 유지하도록 안내 정책을 수립하였습니다.

## 4. 정보 입력/서류 등록 필수값 Validation (유효성 검사) UI 가시성 개선
- **문제점**: 최고관리자가 강제로 해당 회원을 "정상승인" 시킨 일부 테스트 계정의 경우, 실제 등록증 파일이나 필수 데이터가 비어있음에도 불구하고 `status === APPROVED` 조건에 걸려 "🚨 필수첨부 누락" 빨간색 안내 문구가 보이지 않았습니다 (숨김 처리). 그래서 사용자가 서류 보완 경고 창을 누르고 들어와도 무엇이 부족한지 명확히 볼 수 없었습니다.
- **해결책**: 승인 상태(`agencyData.status`)와 상관없이 폼 안의 필수 정보(`reg_num`, `biz_num`, 사진 첨부 등)가 비어 있거나 누락되었다면 **무조건 빨간색 경고 문구가 표시되도록 관련 UI 조건문을 과감하게 수정**했습니다. 

## 5. UI 깜빡임(Flicker) 및 레이아웃 깨짐 버그 수정
- **문제점**: 최고관리자나 회원이 정보를 조회할 때, 서버에서 권한/정보 데이터가 브라우저로 완전히 로딩되기 전(0.5초 경과)에 UI가 텅 빈 기본값('일반회원')으로 우선 그려짐으로써 부동산 탭이 일시적으로 사라지거나 깨져보이는(Flicker) 현상이 있었습니다. 
- **해결책**: `initialFetchDone` 상태 변수를 통해 데이터 패칭(Fetching)이 완료될 때까지 `"회원 정보를 불러오는 중입니다..."` 안내 문구를 먼저 렌더링하고, **안전하게 완료된 뒤 한 번에 UI가 노출되도록** 시각적 안정성을 부여했습니다.

## 6. 최고관리자 헤더 프로필 표시
- **수정사항**: 하드코딩되었던 최고관리자(ADMIN) 헤더를, 다른 일반 및 부동산 회원처럼 DB에서 실제 관리자의 **`이름`**과 **`회원ID(이메일)`**을 패치해 상단 대시보드 좌측상단 배지에 표시되도록 개선했습니다. 추가로, 최고관리자 역시 이 상단 헤더 영역을 클릭하면 곧바로 '정보설정(`?menu=settings`)' 탭으로 이동할 수 있도록 브라우저 라우터 커서 맵핑 및 이벤트 리스너를 붙여두었습니다.

## 7. 슈퍼 관리자 대시보드 내 '댓글 관리' 기능 통합
- **구현**: 기존 뉴스 기사의 댓글을 관리하던 `CommentSection` 컴포넌트를 확장하여, 최상위 슈퍼 관리자의 좌측 사이드바 구조에 `댓글 관리` 메뉴를 추가했습니다.
- **권한 처리**: 관리자 권한(`role="admin"`)일 경우 특정 작성자가 아닌 **전체 활성 댓글**을 로딩하도록 분기 처리하여, 플랫폼 전체의 여론과 사용자 반응을 한 페이지에서 승인/삭제 및 직접 반려할 수 있게 되었습니다.

## 8. 공실톡(GongsilTalk) 중첩 답글(대댓글) 및 권한 필터링 분리
- **DB 스키마 개선**: 기존의 `vacancy_comments` 테이블의 구조적 한계를 보완하기 위해 `parent_id` 외래키(Foreign Key) 컬럼을 추가하는 SQL 마이그레이션을 단행했습니다. 이를 통해 1뎁스의 댓글이 아닌 스레드(Thread) 형태의 다중 대댓글 구조를 완벽하게 지원합니다.
- **카카오톡 스타일 UI**: 관리자의 공실톡 대화창 화면(💬)에서 카카오톡의 '답장' 기능과 동일하게, 어느 메시지에 대한 답변인지를 상단에 인용구(Quotation)로 작게 표시해주어 대화의 문맥(Context)을 쉽게 파악하도록 UI를 정교하게 구현했습니다.

## 9. 공실열람(GongsilClient) 지도 내 뉴스 기사형 인라인 답글 도입
- **문제점**: 그동안 공실열람 지도의 상세 패널에서 댓글을 다는 방식이 말풍선 형태의 카카오톡 스타일이었으며, 답글 버튼을 누르면 입력창 위치가 어색하게 스크롤 되는 불편함이 있었습니다.
- **해결책**:
  - `GongsilClient` 내의 전체 댓글 렌더링 로직을 플랫폼의 뉴스 기사(NewsReadContent)와 동일한 **표준 댓글 리스트 디자인(플랫한 라인형)**으로 전면 리팩토링했습니다.
  - "답글" 버튼 클릭 시, 화면 맨 위나 아래로 튕기지 않고 **타겟팅된 그 댓글 바로 밑 공간에 "인라인(Inline)"으로 쏙 나오도록 답글 작성 폼을 삽입**시켜 뛰어난 UX를 제공합니다. 입력 중 취소 버튼 동작도 직관적으로 연동되었습니다.

## 10. (중요) MY관심공실 (찜/최근조회) DB 연구 저장 최적화 
- **문제점 지적**: 기존의 MY관심공실(찜한 물건 / 최근 본 물건) 기능은 사용자의 브라우저 `localStorage`에만 단순 저장되어, 핸드폰이나 다른 기기로 접속하면 찜 목록이 연동되지 않고 증발해버렸습니다. 또한, 관심 공실마저 현재 "지도에 보이는 영역 안에 있어야만" 필터링되어 리스트에 표시되는 치명적인 버그가 있었습니다.
- **해결책**:
  1. 먼저 `activeCategory === 'wish'`일 때는 지도를 어디로 이동하든 영역 필터링(`mapBounds.contain`)을 무시하고 **모든 찜 목록을 무조건 불러오도록 알고리즘을 우회**시켰습니다.
  2. `sql/create_vacancy_user_data.sql` 마이그레이션을 통해 `vacancy_wishlist` 와 `vacancy_recent_views` 전용 테이블을 구축하고 RLS(Row Level Security) 접근 제한을 걸어 보안을 강화했습니다.
  3. 로그인한 사용자의 경우 로컬 상태와 **데이터베이스(Supabase)를 실시간으로 양방향 동기화(`actions/vacancyUserData.ts`)**하여, 평생 지워지지 않는 북마크 시스템을 견고하게 완성했습니다.

## 11. 게시판 권한 검증 고도화 및 커스텀 알럿 UI 적용
- **문제점**: 
  - 관리자가 설정한 게시판의 읽기, 쓰기, 답변 등급이 실제 프론트엔드 화면(`BoardClient` 등)에 완벽하게 강제되지 않는 백도어(Backdoor) 버그가 존재했습니다.
  - 권한이 없을 경우 브라우저의 투박한 기본 `alert()`가 출력되어 사용자 경험이 저해되었습니다.
  - 관리자 게시판 설정 폼에 더 이상 사용되지 않는 "답변권한", "다운로드권한" 옵션이 시각적으로 방치되어 있었습니다.
- **해결책**:
  1. `permissionCheck.ts` 유틸을 신설하여 회원의 `role` 및 `plan_type`을 결합해 0~5단계 레벨로 정확하게 계층화, 접근 가능 여부를 판별하는 `canAccessBoard` 로직을 구축했습니다.
  2. `BoardReadClient`와 `BoardWriteClient` 단에서 권한 부족 시 목록으로 튕겨내는 가드(Guard) 로직을 추가하고 글쓰기, 답변 버튼을 숨김 처리했습니다.
  3. `BoardClient`에서 접근 권한이 없는 게시물 클릭 시, 밋밋한 브라우저 얼럿 대신 깜찍한 애니메이션이 포함된 블랙 토스트 메시지(Black Toast UI)를 띄워 세련된 경고창 UI로 전면 리뉴얼했습니다. (예: "공실뉴스부동산 회원부터 열람하실 수 있습니다.. 🤍")
  4. 어드민 페이지(`BoardRegisterForm`) 내 불필요한 설정 필드를 제거하여 관리 혼선을 방지했습니다.
