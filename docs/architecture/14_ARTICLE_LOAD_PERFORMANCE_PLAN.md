# 기사 상세뷰 로딩 속도 극대화 (Zero-Latency) 솔루션

기사 리스트는 SSR이 적용되어 즉시 표시되지만, 리스트에서 **기사를 클릭했을 때 0.5초~1초의 지연**이 발생하는 이유는 근본적으로 **데이터 페칭 구조와 라우팅 방식**에 있습니다.

현재 구조는 클릭하는 순간 `Vercel 서버` → `Supabase (DB)` → `Vercel 서버` → `클라이언트 브라우저` 의 왕복 과정(Round-trip)을 반드시 거치게 되어 있습니다.

이를 클릭하는 순간 **0초(Zero-latency)** 에 가깝게 뜨도록 만드는 3단계 솔루션을 제안합니다.

---

## 1단계: URL 구조 변경 (쿼리 파라미터 → 동적 라우팅) 🌟 최우선장려

- **현재 방식:** `/news_read?article_id=27`
- **개선 방식:** `/news/27`

**[왜 빨라지나요?]**
Next.js의 핵심 성능 최적화 기능인 **Prefetching(미리 가져오기)** 때문입니다. 브라우저 화면에 기사 리스트가 보일 때, Next.js는 백그라운드에서 해당 기사들의 데이터를 몰래 미리 받아옵니다. 
하지만 현재처럼 `?article_id=` 형태코드(Search Params)를 사용하면 Next.js가 캐시를 적극적으로 사용하지 않고 매번 새 요청(Dynamic Rendering)으로 처리합니다. 동적 경로(`/news/[id]`)로 바꾸면 클릭 전 이미 데이터가 브라우저에 대기하게 됩니다.

## 2단계: Supabase 쿼리 캐싱 (Data Cache) 적용

현재 코드는 기사를 볼 때마다 매번 Supabase DB에 SELECT 쿼리를 날립니다. 

- **개선 방식:** Next.js의 내장 캐시(`unstable_cache` 또는 `fetch` 캐시)를 적용하여 조회된 기사 데이터를 Vercel 서버의 고속 메모리에 저장합니다.
- **효과:** DB 조회가 생략되어 응답 속도가 최소 3~5배(100ms 이하로) 빨라집니다.
- **동기화:** 기사가 수정되었을 때만 캐시를 초기화(`revalidateTag`) 해주는 로직만 관리자 페이지에 추가하면 됩니다.

## 3단계: ISR 및 사전 렌더링 (Static Generation)

가장 조회수가 높은 "인기 뉴스"나 "최근 등록된 뉴스"는 아예 빌드 단계나 백그라운드에서 완성된 HTML로 미리 구워버리는 방법입니다. (`generateStaticParams` 활용)

- **효과:** 서버 연산 0초, 캐시 조회 0초, 물리적으로 완전한 정적 HTML 파일을 CDN(Edge)에서 바로 내려주므로 **궁극의 속도(SSG)** 를 자랑합니다.

---

## 💡 종합 추천 방안

위의 3가지를 모두 조합하면 네이버나 구글보다도 쾌적한 속도의 뉴스 포털을 만들 수 있습니다. 

현재 당장 체감이 가장 큰 작업은 **1단계(URL 구조 변경)** 와 **2단계(서버 데이터 캐싱 적용)** 입니다.

적용을 원하신다면, **"1, 2단계 적용해줘"** 라고 말씀해 주시면 즉시 코드를 리팩토링하여 체감 속도를 올려드리겠습니다!
