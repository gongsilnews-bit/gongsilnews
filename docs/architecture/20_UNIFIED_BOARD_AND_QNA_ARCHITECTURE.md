# 20. 통합 게시판 시스템 및 확장형 Q&A 아키텍처

## 1. 개요 (Overview)
기존에 공실뉴스 백엔드에 구축되어 있는 **범용 통합 게시판 구조(`boards`, `board_posts`, `board_comments`)**를 적극적으로 재사용하여, 새로운 기능(예: 특강 Q&A, 부동산 미니 홈페이지 게시판)이 추가될 때마다 별도의 테이블이나 복잡한 API를 파편화하여 생성하지 않고 하나의 파이프라인으로 처리하기 위한 아키텍처 전략입니다.

## 2. 핵심 설계: `reference_id` 매핑 패턴
통합 게시판 시스템이 다양한 도메인(강의, 부동산 등)에 기생하여 작동할 수 있도록 `board_posts` 테이블에 단일 식별자 컬럼을 추가합니다.

*   **추가 컬럼**: `reference_id` (VARCHAR / UUID 호환)
*   **역할**: 특정 게시글이 '어느 강의' 혹은 '어느 부동산'에 종속되는지를 가리키는 포인터 역할을 수행합니다.

이 패턴을 통해 `board_id`(대분류)와 `reference_id`(소분류/종속성)의 조합만으로 무한한 형태의 게시판 파생이 가능해집니다.

---

## 3. 적용 시나리오 1: 부동산 특강 Q&A (Study Q&A)
수강생들이 특강을 들으며 질문을 남기고, 강사가 답변을 다는 시스템입니다.

*   **데이터 매핑**:
    *   `board_id`: `'lecture_qna'` (고정)
    *   `reference_id`: `[현재_수강중인_강의의_ID]` (예: 미국증시 강의 UUID)
*   **프론트엔드 연동 (`study_watch`)**:
    *   Q&A 탭 클릭 시, 전체 게시글 중 `board_id === 'lecture_qna'` 이고 `reference_id === [강의ID]` 인 데이터만 `getBoardPosts` API로 불러옵니다.
*   **답변 시스템**:
    *   기존에 완벽히 구현된 `board_comments` 테이블을 활용해, 해당 포스트(질문)에 달린 댓글(Comment)을 강사의 답변(Answer)으로 렌더링합니다.

---

## 4. 적용 시나리오 2: 부동산 전용 미니 홈페이지 (Premium Realty 연계)
**프리미엄 부동산 결제 시스템(19번 문서 Phase 4)**에서 기획된 `gongsilnews.com/realty/[상호명]` 미니 홈페이지 내부의 전용 게시판 시스템입니다.

*   **데이터 매핑**:
    *   `board_id`: `'realty_notice'` (공지사항) 또는 `'realty_qna'` (고객 문의)
    *   `reference_id`: `[중개사_ID]` 또는 `[상호명]`
*   **프론트엔드 연동 (`/realty/[상호명]`)**:
    *   방문자가 특정 부동산의 미니 홈페이지에 들어오면, URL에서 추출한 `[상호명]`을 쿼리에 넣어 해당 부동산 사장님이 작성한 공지사항이나 고객 문의글만 독립적으로 불러옵니다.
*   **독립성 보장**:
    *   데이터베이스 테이블을 하나만 사용하지만, UI 상으로는 각 부동산 중개사들이 완벽히 독립된 자신만의 게시판을 소유한 것처럼 보이게 됩니다.

---

## 5. 아키텍처 기대 효과 및 장점

1.  **유지보수 리소스 절감 (DRY 원칙)**
    *   게시판 생성, 수정, 삭제, 첨부파일(`board_attachments`), 조회수 증가, 댓글 작성 등 기존에 만들어진 API Action(`src/app/actions/board.ts`) 코드를 100% 재활용하므로 백엔드 개발 공수가 획기적으로 줄어듭니다.
2.  **단일화된 어드민(Admin) 관리 경험**
    *   관리자 대시보드에 '통합 게시판 관리' 메뉴 하나만 구축해 두면, [특강 Q&A], [자유게시판], [부동산 공지사항] 등 플랫폼 내에서 발생하는 모든 사용자 게시물을 한 페이지에서 필터링하고 모니터링/제재할 수 있습니다.
3.  **높은 미래 확장성 (Future-proof)**
    *   추후 새로운 커뮤니티 카테고리가 100개 생겨나더라도 DB 스키마 수정 없이, 단지 `boards` 테이블에 `board_id` 레코드만 1개 추가하면 즉시 서비스 론칭이 가능합니다.
