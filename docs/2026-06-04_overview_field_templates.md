## 1. 현재 문제

Page 1 (개요) 테이블이 **빌딩/상가건물 기준 9칸으로 하드코딩**되어 있어,
아파트·오피스텔·토지 등 다른 물건 유형에서는 부적합한 항목이 노출됨.

```
현재 하드코딩 필드 (빌딩 전용):
소재지 / 용도지역 / 대지면적 / 연면적 / 건물규모 / 주용도 / 주차대수 / 승강기 / 준공연도
```

---

## 2. 설계 원칙

- 테이블 구조: **9칸 (물건 정보) + 1칸 (가격)** = 총 10행 고정
- **물건 유형**에 따라 9칸의 필드명이 바뀜
- **거래 유형**(매매/전세/월세)에 따라 가격행 라벨 + 일부 필드(9번) 교체
- 10번째 행(가격)은 거래유형에 따라 자동 변환 (이미 구현됨)

### 값이 없는 필드 처리

- 물건 데이터에 해당 항목이 없을 경우 → **필드명(라벨)은 표시, 값은 공란**
- 사용자가 캔버스에서 클릭하여 **직접 입력** 가능
- 향후 AI(Gemini)가 상세설명에서 추출하여 **자동 채움** 가능

### 거래유형별 차이점 (9번 칸 분기)

| 거래유형 | 9번 칸 | 비고 |
|:---------|:-------|:-----|
| 매매     | 준공연도 | 매수자: 건물 나이 중요 |
| 전세/월세/단기 | 관리비 | 임차인: 월 비용 중요 |

> ⚠️ 나머지 1~8번 칸은 거래유형과 무관하게 물건유형에 의해서만 결정됨

## 4. 유형 자동 판별 규칙

공실뉴스 데이터의 물건 유형 문자열을 기반으로 템플릿 매핑:

| 데이터 값 | 적용 템플릿 |
|:----------|:-----------|
| `아파트` | 아파트 |
| `오피스텔` | 오피스텔 |
| `빌딩`, `건물`, `상가건물` | 빌딩/상가건물 |
| `상가`, `점포` | 상가/점포 |
| `사무실` | 사무실 |
| `토지`, `나대지` | 토지 |
| `단독`, `다가구`, `다세대` | 단독/다가구 |
| `원룸`, `투룸`, `쓰리룸` | 원룸/투룸 |
| 기타 / 미지정 | 빌딩/상가건물 (기본값) |

## 6. 구현 범위

### 변경 대상 파일

| 파일 | 변경 내용 |
|:-----|:---------|
| `propertyTemplates.ts` (신규) | 8가지 물건유형 × 거래유형별 9칸 필드 정의 |
